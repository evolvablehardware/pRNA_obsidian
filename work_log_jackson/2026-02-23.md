	- Not sure how to approach this from a design standpoint, currently just switching inside existing pulsecountfitnessfunction. Right now I think the FitnessFunction is trying to both act as a way to measure from the device in addition to actually calculating the fitness (right now sensitive and tolerant done from same class). This is sort of complicated by the fact that most fitness metrics are specific to the type of experiment, I don't think its possible to make one interface for all potential metrics. If we decide to only support icefarm, its probably best to completely remove the current fitnessfunction in one more aimed at just calculating fitness, as the measurement stuff is taken care of by icefarm. If we want to leave room to run without icefarm, probably need to keep it around in its current form, but separate the metric aspect so it can be used across both evaluation methods, as the remote evaluation option would need a separate fitnessfunction
